# Product Plan

이 문서는 `openapi-workflow`가 무엇을 만들려는 도구인지, 어떤 문제를 풀고 어떤 범위를 목표로 하는지 정의합니다.

## 한 줄 정의

`openapi-workflow`는 OpenAPI 스펙을 입력으로 받아, 프론트엔드 프로젝트에 맞는 TypeScript DTO/API 파일 생성까지 단계적으로 지원하는 CLI 워크플로우입니다.

## 문제 정의

프론트엔드 프로젝트에서 OpenAPI 기반 코드를 만들 때 보통 아래 문제가 반복됩니다.

- OpenAPI에서 DTO, API 초안을 만드는 작업은 반복적이지만 프로젝트마다 결과 형식이 다릅니다.
- 프로젝트 구조, import 경로, 응답 래퍼, query 규칙 같은 컨벤션은 스펙만으로 결정할 수 없습니다.
- 이 차이를 코드에 바로 하드코딩하면 재사용성이 떨어지고, 사람 손으로만 맞추면 비용이 큽니다.

즉 이 도구가 풀고 싶은 문제는 다음과 같습니다.

- 스펙만으로 결정 가능한 부분은 코드로 deterministic 하게 생성하고
- 프로젝트 의존적인 부분은 규칙 문서로 분리하고
- 두 결과를 결합해 현재 프로젝트 컨벤션에 맞는 파일을 생성하는 것

## 제품 목표

이 도구의 핵심 목표는 아래 3단계입니다.

### 1. 결정 가능한 부분의 코드 생성

OpenAPI 스펙만으로 결정 가능한 정보를 사용해 아래 산출물을 자동 생성합니다.

- endpoint catalog
- review 문서
- raw TypeScript DTO
- raw TypeScript API helper

이 단계의 목적은 “스펙 해석 결과를 사람이 검토 가능하고 재생성 가능하게 만드는 것”입니다.

### 2. 프로젝트 규칙의 문서화

현재 적용하려는 프론트엔드 프로젝트를 분석해 아래 정보를 문서/설정으로 명시화합니다.

- 디렉터리 구조
- 공통 API 호출 함수 경로와 이름
- 응답 래퍼 타입
- 페이징 타입
- query/body/multipart 처리 규칙
- 필요 시 path 상수, URL 매핑, 네이밍 규칙

이 단계의 목적은 “프로젝트 의존적인 결정을 코드 밖의 명시적 규칙으로 분리하는 것”입니다.

### 3. 현재 프로젝트 컨벤션에 맞는 파일 생성

1단계에서 만든 deterministic 산출물과 2단계에서 문서화한 규칙을 함께 사용해, 현재 프로젝트에서 바로 검토 가능한 후보 파일을 생성합니다.

- 프로젝트 맞춤 DTO 파일
- 프로젝트 맞춤 API 파일
- 적용 대상 경로 정보
- 생성 결과 manifest / summary

이 단계의 목적은 “사람이 마지막 검토 후 반영할 수 있는 프로젝트 맞춤 산출물”을 만드는 것입니다.

## 핵심 원칙

- `apply` 전까지는 실제 앱 코드를 직접 수정하지 않습니다.
- 스펙 기반 변환과 프로젝트 기반 변환을 분리합니다.
- 규칙은 사람이 읽고 수정할 수 있는 파일이어야 합니다.
- 같은 입력이면 같은 결과가 나와야 합니다.
- 사람 검토 단계를 생략하지 않는 것을 기본값으로 둡니다.

## 주요 사용자

- OpenAPI 기반 프론트엔드 연동 코드를 빠르게 만들고 싶은 개발자
- 여러 서비스 프로젝트에 동일 워크플로우를 재사용하고 싶은 팀
- AI와 사람이 함께 코드 생성 규칙을 정리하고 싶은 유지보수자

## 기본 사용 흐름

1. `init`
2. `refresh`
3. `rules`
4. `project`
5. `apply`

세부 의미:

- `refresh`
  - OpenAPI 다운로드, catalog 생성, review 산출물 생성
- `rules`
  - 현재 프로젝트 구조를 분석하고 규칙 파일 초안 생성
- `project`
  - 규칙 파일을 읽어 프로젝트 맞춤 후보 코드 생성
- `apply`
  - 후보 코드를 실제 `src`에 반영

## 비목표

현재 이 도구의 비목표는 아래와 같습니다.

- 모든 프론트엔드 프레임워크를 완전 자동 지원
- 사람 검토 없이 곧바로 기존 코드베이스를 안전하게 변경
- 프로젝트 컨벤션을 100% 추론만으로 완성
- 백엔드 스펙 오류까지 자동 보정
- 런타임 클라이언트 전체 구현까지 완전 대체

## 성공 기준

이 도구는 아래가 가능할 때 목표에 부합한다고 봅니다.

- OpenAPI 스펙만으로 raw DTO/API를 반복 가능하게 생성할 수 있다.
- 프로젝트 특화 규칙을 문서와 설정 파일로 분리할 수 있다.
- 규칙 파일을 바꾸면 프로젝트 맞춤 결과물도 예측 가능하게 바뀐다.
- `apply` 전 단계에서 충분한 리뷰가 가능하다.
- 서로 다른 서비스 프로젝트에서도 동일한 도구 구조를 재사용할 수 있다.

## 현재 제품 상태

현재 저장소는 위 목표를 향한 standalone CLI prototype 단계입니다.

- 1단계는 상당 부분 구현됨
- 2단계는 특정 프로젝트 구조를 가정한 분석기 수준으로 구현됨
- 3단계는 프로젝트 맞춤 후보 생성 흐름은 있으나, 규칙 반영 범위는 더 확장 필요

즉 방향은 맞지만, 아직 “범용 프로젝트 적응형 생성기”까지는 완성되지 않았습니다.
