# External Report Review

이 문서는 외부 설계 보고서와 그에 대한 검토 의견을 바탕으로, 현재 `openapi-workflow`가 무엇을 채택하고 무엇을 후순위로 둘지 정리한 내부 결정 문서입니다.

## 목적

외부 보고서는 품질이 높고 설계 범위도 넓습니다.

하지만 현재 프로젝트는 아직 prototype 단계이므로, 보고서 전체를 곧바로 구현 범위로 받아들이기보다 아래 기준으로 정리할 필요가 있습니다.

- 지금 당장 채택할 것
- 구조는 받아들이되 MVP에서는 제외할 것
- 장기 과제로만 남길 것

이 문서는 그 판단을 남기기 위한 문서입니다.

## 요약 결론

- 외부 보고서는 설계 문서로서 매우 우수하다.
- 현재 프로젝트의 방향성은 보고서와 대체로 일치한다.
- 다만 현재 프로젝트는 보고서의 “축소된 첫 구현”으로 봐야 한다.
- 따라서 보고서를 기준 설계서로 삼되, 1차 구현 범위는 더 좁혀야 한다.

## 현재 프로젝트와의 관계

현재 프로젝트는 이미 아래 구조를 갖고 있습니다.

- 스펙 기반 raw 산출물 생성
- 프로젝트 규칙 분석 및 문서화
- 규칙 반영 후보 코드 생성

이는 외부 보고서가 강조한 아래 구조와 맞닿아 있습니다.

- parse / resolve
- normalize / intermediate representation
- generate
- validate / format

차이는 범위입니다.

- 외부 보고서:
  - OpenAPI 2.0 / 3.0 / 3.1
  - React / Next / Vue3
  - Ajv / Zod
  - watch / incremental / security / CI
- 현재 프로젝트:
  - OAS3 스타일 중심 구현
  - 특정 TypeScript 프론트엔드 구조를 가정한 규칙 분석
  - review `schema.ts` + project 엔드포인트별 DTO/API 생성 중심

즉 방향은 맞지만, 지원 범위와 제품화 밀도는 아직 차이가 큽니다.

## 채택할 판단

### 1. 문제 정의는 그대로 채택

가장 중요한 문제 정의는 유지합니다.

- 이 도구는 단순 타입 생성기가 아니다.
- OpenAPI 스펙만으로 결정 가능한 부분과 프로젝트 의존적인 부분을 분리해야 한다.
- 최종 목표는 “프로젝트 컨벤션 적응형 generator”이다.

이는 현재 [Product Plan](05-product-plan.md)과도 일치합니다.

### 2. 파이프라인 구조는 채택

외부 보고서가 제시한 파이프라인 사고방식은 채택합니다.

- 입력 로드
- 스펙 해석
- 정규화
- 생성
- 검증/포맷팅

현재 구현이 이 구조를 완전히 갖춘 것은 아니지만, 앞으로의 리팩터링 방향은 이 기준을 따릅니다.

### 3. AI는 추천 도구로만 제한

외부 보고서와 검토 의견 모두 같은 방향을 제시합니다.

- AI는 규칙 추천, 네이밍 제안, 설정 초안 생성에만 사용
- 최종 생성은 config + code path 로 deterministic 하게 수행

즉 “AI가 직접 생성물을 결정”하는 구조는 채택하지 않습니다.

## MVP에서 바로 채택할 범위

### 타겟

- React / Next.js + TypeScript

### 입력 범위

- OpenAPI 3.0 / 3.1 JSON

### 출력 범위

- DTO TypeScript 타입
- fetch 기반 API client
- 프로젝트 규칙 기반 output path / naming / import 조정

### 컨벤션 반영 방식

- 사람 또는 AI가 수정 가능한 config 우선
- 프로젝트 분석은 보조 수단
- 생성 결과는 항상 규칙 파일이 지배

## MVP에서 제외할 범위

아래 항목은 설계상 중요하지만 1차 구현 범위에서는 제외합니다.

- OpenAPI 2.0 직접 지원
- Vue3 전용 출력
- Ajv / Zod 런타임 검증 생성
- watch / incremental generation
- SSRF hardening 포함 원격 fetch 보안 정책 완비
- React Query / SWR / Vue composable 자동 생성

이 항목들은 “불필요”해서가 아니라 “지금 넣으면 범위가 과도해지기 때문”에 제외합니다.

## 기술 방향 결정

외부 보고서의 비교 결과와 현재 구현 상태를 함께 고려했을 때, 1차 기술 방향은 아래로 정리합니다.

### 1차 구현 권장 방향

- 타입 생성:
  - `openapi-typescript` 활용 가능성 검토
- HTTP 클라이언트:
  - `openapi-fetch` 또는 얇은 custom fetch wrapper
- 프로젝트 맞춤 출력:
  - 자체 config + 자체 generator 계층

### 장기 방향

- 필요 시 stronger parser + IR + custom generator 코어로 확장
- 또는 OpenAPI Generator 템플릿 전략과 비교해 병행 검토

즉 “모든 것을 처음부터 직접 구현”보다는, 활용 가능한 기반 도구는 쓰되 프로젝트 적응 계층은 직접 소유하는 방향을 채택합니다.

## 현재 프로젝트에 주는 영향

이 문서를 기준으로 보면 현재 프로젝트의 우선순위는 더 명확해집니다.

### 가장 먼저 해야 할 일

- 규칙 파일이 생성 결과를 더 강하게 지배하게 만들기
- output path / naming / import 규칙을 config화하기
- `rules`의 특정 구조 의존성을 완화하기

### 그 다음 해야 할 일

- schema / config validation 추가
- fixture 기반 회귀 테스트 추가
- `apply` 안전장치 추가

즉 현재 프로젝트는 “더 많은 기능 추가”보다 “현재 구조를 제품 설계에 맞게 정리하는 것”이 우선입니다.

## 최종 판단

외부 보고서는 현재 프로젝트보다 “한 단계 위의 설계 기준”으로 채택할 가치가 있습니다.

다만 지금 당장의 구현 기준은 아래처럼 정리합니다.

- 기준 설계:
  - 외부 보고서
- 실제 MVP 범위:
  - 더 작고 더 실전적으로 제한
- 현재 프로젝트의 역할:
  - 그 축소된 MVP를 먼저 구현하는 기반

즉 이 프로젝트는 외부 보고서와 경쟁하는 것이 아니라, 그 보고서를 현실적인 범위로 줄여 실제 제품으로 옮기는 작업이라고 보는 것이 맞습니다.
